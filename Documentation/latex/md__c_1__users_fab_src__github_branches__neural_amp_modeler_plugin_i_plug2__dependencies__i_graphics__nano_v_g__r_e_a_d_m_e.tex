Nano\+VG is small antialiased vector graphics rendering library for Open\+GL. It has lean A\+PI modeled after H\+T\+M\+L5 canvas A\+PI. It is aimed to be a practical and fun toolset for building scalable user interfaces and visualizations.\hypertarget{md__c_1__users_fab_src__github_branches__neural_amp_modeler_plugin_i_plug2__dependencies__i_graphics__nano_v_g__r_e_a_d_m_e_autotoc_md1337}{}\doxysection{Screenshot}\label{md__c_1__users_fab_src__github_branches__neural_amp_modeler_plugin_i_plug2__dependencies__i_graphics__nano_v_g__r_e_a_d_m_e_autotoc_md1337}
\hypertarget{md__c_1__users_fab_src__github_branches__neural_amp_modeler_plugin_i_plug2__dependencies__i_graphics__nano_v_g__r_e_a_d_m_e_autotoc_md1338}{}\doxysection{Usage}\label{md__c_1__users_fab_src__github_branches__neural_amp_modeler_plugin_i_plug2__dependencies__i_graphics__nano_v_g__r_e_a_d_m_e_autotoc_md1338}
The Nano\+VG A\+PI is modeled loosely on H\+T\+M\+L5 canvas A\+PI. If you know canvas, you\textquotesingle{}re up to speed with Nano\+VG in no time.\hypertarget{md__c_1__users_fab_src__github_branches__neural_amp_modeler_plugin_i_plug2__dependencies__i_graphics__nano_v_g__r_e_a_d_m_e_autotoc_md1339}{}\doxysubsection{Creating drawing context}\label{md__c_1__users_fab_src__github_branches__neural_amp_modeler_plugin_i_plug2__dependencies__i_graphics__nano_v_g__r_e_a_d_m_e_autotoc_md1339}
The drawing context is created using platform specific constructor function. If you\textquotesingle{}re using the Open\+GL 2.\+0 back-\/end the context is created as follows\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{preprocessor}{\#define NANOVG\_GL2\_IMPLEMENTATION   // Use GL2 implementation.}}
\DoxyCodeLine{\textcolor{preprocessor}{\#include "nanovg\_gl.h"}}
\DoxyCodeLine{...}
\DoxyCodeLine{struct \mbox{\hyperlink{struct_n_v_gcontext}{NVGcontext}}* vg = nvgCreateGL2(NVG\_ANTIALIAS | NVG\_STENCIL\_STROKES);}
\end{DoxyCode}


The first parameter defines flags for creating the renderer.


\begin{DoxyItemize}
\item {\ttfamily N\+V\+G\+\_\+\+A\+N\+T\+I\+A\+L\+I\+AS} means that the renderer adjusts the geometry to include anti-\/aliasing. If you\textquotesingle{}re using M\+S\+AA, you can omit this flags.
\item {\ttfamily N\+V\+G\+\_\+\+S\+T\+E\+N\+C\+I\+L\+\_\+\+S\+T\+R\+O\+K\+ES} means that the render uses better quality rendering for (overlapping) strokes. The quality is mostly visible on wider strokes. If you want speed, you can omit this flag.
\end{DoxyItemize}

Currently there is an Open\+GL back-\/end for Nano\+VG\+: \href{/src/nanovg_gl.h}{\texttt{ nanovg\+\_\+gl.\+h}} for Open\+GL 2.\+0, Open\+GL ES 2.\+0, Open\+GL 3.\+2 core profile and Open\+GL ES 3. The implementation can be chosen using a define as in above example. See the header file and examples for further info.

{\itshape N\+O\+TE\+:} The render target you\textquotesingle{}re rendering to must have stencil buffer.\hypertarget{md__c_1__users_fab_src__github_branches__neural_amp_modeler_plugin_i_plug2__dependencies__i_graphics__nano_v_g__r_e_a_d_m_e_autotoc_md1340}{}\doxysubsection{Drawing shapes with Nano\+VG}\label{md__c_1__users_fab_src__github_branches__neural_amp_modeler_plugin_i_plug2__dependencies__i_graphics__nano_v_g__r_e_a_d_m_e_autotoc_md1340}
Drawing a simple shape using Nano\+VG consists of four steps\+: 1) begin a new shape, 2) define the path to draw, 3) set fill or stroke, 4) and finally fill or stroke the path.


\begin{DoxyCode}{0}
\DoxyCodeLine{nvgBeginPath(vg);}
\DoxyCodeLine{nvgRect(vg, 100,100, 120,30);}
\DoxyCodeLine{nvgFillColor(vg, nvgRGBA(255,192,0,255));}
\DoxyCodeLine{nvgFill(vg);}
\end{DoxyCode}


Calling {\ttfamily nvg\+Begin\+Path()} will clear any existing paths and start drawing from blank slate. There are number of number of functions to define the path to draw, such as rectangle, rounded rectangle and ellipse, or you can use the common move\+To, line\+To, bezier\+To and arc\+To A\+PI to compose the paths step by step.\hypertarget{md__c_1__users_fab_src__github_branches__neural_amp_modeler_plugin_i_plug2__dependencies__i_graphics__nano_v_g__r_e_a_d_m_e_autotoc_md1341}{}\doxysubsection{Understanding Composite Paths}\label{md__c_1__users_fab_src__github_branches__neural_amp_modeler_plugin_i_plug2__dependencies__i_graphics__nano_v_g__r_e_a_d_m_e_autotoc_md1341}
Because of the way the rendering backend is build in Nano\+VG, drawing a composite path, that is path consisting from multiple paths defining holes and fills, is a bit more involved. Nano\+VG uses even-\/odd filling rule and by default the paths are wound in counter clockwise order. Keep that in mind when drawing using the low level draw A\+PI. In order to wind one of the predefined shapes as a hole, you should call {\ttfamily nvg\+Path\+Winding(vg, N\+V\+G\+\_\+\+H\+O\+L\+E)}, or {\ttfamily nvg\+Path\+Winding(vg, N\+V\+G\+\_\+\+C\+W)} {\itshape after} defining the path.


\begin{DoxyCode}{0}
\DoxyCodeLine{nvgBeginPath(vg);}
\DoxyCodeLine{nvgRect(vg, 100,100, 120,30);}
\DoxyCodeLine{nvgCircle(vg, 120,120, 5);}
\DoxyCodeLine{nvgPathWinding(vg, NVG\_HOLE);   // Mark circle as a hole.}
\DoxyCodeLine{nvgFillColor(vg, nvgRGBA(255,192,0,255));}
\DoxyCodeLine{nvgFill(vg);}
\end{DoxyCode}

\begin{DoxyItemize}
\item make sure you have created Nano\+VG context using one of the {\ttfamily nvg\+Createxxx()} calls
\item make sure you have initialised Open\+GL with {\itshape stencil buffer}
\item make sure you have cleared stencil buffer
\item make sure all rendering calls happen between {\ttfamily nvg\+Begin\+Frame()} and {\ttfamily nvg\+End\+Frame()}
\item to enable more checks for Open\+GL errors, add {\ttfamily N\+V\+G\+\_\+\+D\+E\+B\+UG} flag to {\ttfamily nvg\+Createxxx()}
\item if the problem still persists, please report an issue!

The Open\+GL back-\/end touches following states\+:

When textures are uploaded or updated, the following pixel store is set to defaults\+: {\ttfamily G\+L\+\_\+\+U\+N\+P\+A\+C\+K\+\_\+\+A\+L\+I\+G\+N\+M\+E\+NT}, {\ttfamily G\+L\+\_\+\+U\+N\+P\+A\+C\+K\+\_\+\+R\+O\+W\+\_\+\+L\+E\+N\+G\+TH}, {\ttfamily G\+L\+\_\+\+U\+N\+P\+A\+C\+K\+\_\+\+S\+K\+I\+P\+\_\+\+P\+I\+X\+E\+LS}, {\ttfamily G\+L\+\_\+\+U\+N\+P\+A\+C\+K\+\_\+\+S\+K\+I\+P\+\_\+\+R\+O\+WS}. Texture binding is also affected. Texture updates can happen when the user loads images, or when new font glyphs are added. Glyphs are added as needed between calls to {\ttfamily nvg\+Begin\+Frame()} and {\ttfamily nvg\+End\+Frame()}.

The data for the whole frame is buffered and flushed in {\ttfamily nvg\+End\+Frame()}. The following code illustrates the Open\+GL state touched by the rendering code\+:
