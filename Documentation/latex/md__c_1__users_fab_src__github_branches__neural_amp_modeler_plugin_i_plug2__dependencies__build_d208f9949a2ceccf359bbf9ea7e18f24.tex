{\ttfamily faust2smartkeyb} is a tool to generate ready-\/to-\/use musical Android and i\+OS applications using the \href{http://faust.grame.fr}{\texttt{ Faust programming language}}. Unlike \href{https://ccrma.stanford.edu/~rmichon/faust2android/}{\texttt{ {\ttfamily faust2android}}} and {\ttfamily faust2ios}, {\ttfamily faust2smartkeyb} ignores the standard user interface (\mbox{\hyperlink{class_u_i}{UI}}) declared in the \mbox{\hyperlink{struct_faust}{Faust}} code (e.\+g., {\ttfamily hslider}, {\ttfamily button}, etc.), and replaces it by a {\ttfamily Smart\+Keyboard} \mbox{\hyperlink{class_u_i}{UI}}.

The {\ttfamily Smart\+Keyboard} \mbox{\hyperlink{class_u_i}{UI}} allows to implement a wide range of controllers (basic keyboards, isomorphic keyboards, pads, X/Y controllers, etc.) on a touch-\/screen and can be configured directly in the \mbox{\hyperlink{struct_faust}{Faust}} code using the {\ttfamily Smart\+Keyboard} metadata.

This documentation demonstrates how to use {\ttfamily faust2smartkeyb} and provides \href{\#additional-resources}{\texttt{ a series of links to tutorials on how to turn mobile devices into musical instruments}}.

{\ttfamily faust2smartkeyb} is part of the \href{https://github.com/grame-cncm/faust}{\texttt{ Faust distribution}} and will be automatically installed on your system with the latest version of \mbox{\hyperlink{struct_faust}{Faust}}.

{\ttfamily faust2smartkeyb} uses {\ttfamily faust2api} internally. Thus, \mbox{\hyperlink{struct_faust}{Faust}} codes provided to {\ttfamily faust2smartkeyb} must respect the same standards as for {\ttfamily faust2api} (check the \href{https://ccrma.stanford.edu/~rmichon/faust2api/}{\texttt{ {\ttfamily faust2api} documentation}} for more information about this).

The following code is similar to the one presented in the \href{https://ccrma.stanford.edu/~rmichon/faust2api/}{\texttt{ {\ttfamily faust2api} documentation}} in the \href{https://ccrma.stanford.edu/~rmichon/faust2api/\#midi-enabled-polyphonic-object}{\texttt{ M\+I\+DI Enabled Polyphonic Object}}\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{import("stdfaust.lib");}
\DoxyCodeLine{freq = nentry("freq",200,40,2000,0.01);}
\DoxyCodeLine{gain = nentry("gain",1,0,1,0.01);}
\DoxyCodeLine{gate = button("gate");}
\DoxyCodeLine{cutoff = 1000;}
\DoxyCodeLine{envelope = gate*gate : si.smoo;}
\DoxyCodeLine{process = os.sawtooth(freq)*envelope : fi.lowpass(3,cutoff) : *(0.25) <: \_,\_;}
\end{DoxyCode}


It implements a simple synthesizer based on a filtered sawtooth wave and it is polyphony-\/compatible thanks to its {\ttfamily freq}, {\ttfamily gain}, and {\ttfamily gate} parameters. Note, that the gain of the synth is scaled. This is done to prevent clicking if several polyphony voices are played at the same time. When using an effect with a synth, it\textquotesingle{}s better practice to carry out this type of gain scaling in the effect section to save up computation (see \href{\#effect}{\texttt{ {\ttfamily effect}}}).

\DoxyCodeLine{    // configuration keys}
\DoxyCodeLine{\}";}

We assume that you followed the steps described in the \href{\#setting-up-your-system}{\texttt{ Setting-\/\+Up Your System}} section before reading this. Only 2 arguments are required to use {\ttfamily faust2smartkeyb}\+: the target platform (i\+OS or Android) and a \mbox{\hyperlink{struct_faust}{Faust}} code\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{faust2smartkeyb -\/ios mySynth.dsp}
\end{DoxyCode}


will compile {\ttfamily my\+Synth.\+dsp} into an i\+OS app ({\ttfamily my\+Synth.\+app}) with a {\ttfamily Smart\+Keyboard} interface and


\begin{DoxyCode}{0}
\DoxyCodeLine{faust2smartkeyb -\/android mySynth.dsp}
\end{DoxyCode}


will do the same for Android ({\ttfamily my\+Synth.\+apk}).

While this should be very smooth on Android if your followed the steps in \href{\#setting-up-your-system}{\texttt{ Setting-\/\+Up Your System}}, this will probably not work on i\+OS. Why? Because the bundle identifier associated with the template app used to compile a {\ttfamily Smart\+Keyboard} app for i\+OS is not yours. So unless you change it directly in the source code of the \mbox{\hyperlink{struct_faust}{Faust}} distribution ({\ttfamily /architecture/smart\+Keyboard/i\+O\+S/\+Faust.xcodeproj}) and re-\/install \mbox{\hyperlink{struct_faust}{Faust}}, it will not work (yes, we know, we know... \+:( ).

However, in practice, you\textquotesingle{}ll rarely want to compile directly your \mbox{\hyperlink{struct_faust}{Faust}} code into an app package. Instead, you might prefer to create an Android Studio on an Xcode project associated to your \mbox{\hyperlink{struct_faust}{Faust}} code and update it when necessary. This is a much better solution because compiling an app from scratch every time (especially on Android) takes a lot of time ($>$1 minute in most cases). To do this, you can use the \href{\#source}{\texttt{ {\ttfamily -\/source}}} option in combination with \href{\#reuse}{\texttt{ {\ttfamily -\/reuse}}}\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{faust2smartkeyb -\/android -\/source -\/reuse mySynth.dsp}
\end{DoxyCode}


In that case, {\ttfamily faust2smartkeyb} will not compile {\ttfamily my\+Synth.\+dsp} to an app but will create a folder called {\ttfamily faustsmartkeyb.\+my\+Synth} in the current folder containing an Android Studio project. Every time the previous command will be run, the portion of the app source code corresponding to {\ttfamily my\+Synth.\+dsp} will be updated (if we only used {\ttfamily -\/source} without {\ttfamily -\/reuse} then {\ttfamily faustsmartkeyb.\+my\+Synth} would be erased and re-\/created). The same steps can be followed when using {\ttfamily -\/ios}.

The app can be easily installed on your device from Xcode or Android Studio. On i\+OS, bundle identifier issues can be fixed directly in the Xcode project contained in {\ttfamily faustsmartkeyb.\+my\+Synth} which is more convenient than doing it in the \mbox{\hyperlink{struct_faust}{Faust}} source code.
