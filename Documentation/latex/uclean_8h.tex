{\footnotesize\ttfamily \mbox{\hyperlink{umachine_8h_a676114b1a64bb7c2de15c919a00b28df}{U\+\_\+\+C\+A\+PI}} void U\+\_\+\+E\+X\+P\+O\+R\+T2 u\+\_\+cleanup (\begin{DoxyParamCaption}\item[{void}]{ }\end{DoxyParamCaption})}

Clean up the system resources, such as allocated memory or open files, used in all I\+CU libraries. This will free/delete all memory owned by the I\+CU libraries, and return them to their original load state. All open I\+CU items (collators, resource bundles, converters, etc.) must be closed before calling this function, otherwise I\+CU may not free its allocated memory (e.\+g. close your converters and resource bundles before calling this function). Generally, this function should be called once just before an application exits. For applications that dynamically load and unload the I\+CU libraries (relatively uncommon), \mbox{\hyperlink{uclean_8h_a02cee2494284801dcc7f8bf7f12f0bcd}{u\+\_\+cleanup()}} should be called just before the library unload. 

\mbox{\hyperlink{uclean_8h_a02cee2494284801dcc7f8bf7f12f0bcd}{u\+\_\+cleanup()}} also clears any I\+CU heap functions, mutex functions or trace functions that may have been set for the process. ~\newline
 This has the effect of restoring I\+CU to its initial condition, before any of these override functions were installed. Refer to \mbox{\hyperlink{uclean_8h_ae5834e6f4eaa14c1929d30f047da93a8}{u\+\_\+set\+Memory\+Functions()}}, u\+\_\+set\+Mutex\+Functions and \mbox{\hyperlink{utrace_8h_a2549611e2c929a2470218d1f6342fb29}{utrace\+\_\+set\+Functions()}}. If I\+CU is to be reinitialized after calling \mbox{\hyperlink{uclean_8h_a02cee2494284801dcc7f8bf7f12f0bcd}{u\+\_\+cleanup()}}, these runtime override functions will need to be set up again if they are still required. 

\mbox{\hyperlink{uclean_8h_a02cee2494284801dcc7f8bf7f12f0bcd}{u\+\_\+cleanup()}} is not thread safe. All other threads should stop using I\+CU before calling this function. 

Any open I\+CU items will be left in an undefined state by \mbox{\hyperlink{uclean_8h_a02cee2494284801dcc7f8bf7f12f0bcd}{u\+\_\+cleanup()}}, and any subsequent attempt to use such an item will give unpredictable results. 

After calling \mbox{\hyperlink{uclean_8h_a02cee2494284801dcc7f8bf7f12f0bcd}{u\+\_\+cleanup()}}, an application may continue to use I\+CU by calling \mbox{\hyperlink{uclean_8h_a4774045efa17dd592f69f6988039a672}{u\+\_\+init()}}. An application must invoke \mbox{\hyperlink{uclean_8h_a4774045efa17dd592f69f6988039a672}{u\+\_\+init()}} first from one single thread before allowing other threads call \mbox{\hyperlink{uclean_8h_a4774045efa17dd592f69f6988039a672}{u\+\_\+init()}}. All threads existing at the time of the first thread\textquotesingle{}s call to \mbox{\hyperlink{uclean_8h_a4774045efa17dd592f69f6988039a672}{u\+\_\+init()}} must also call \mbox{\hyperlink{uclean_8h_a4774045efa17dd592f69f6988039a672}{u\+\_\+init()}} themselves before continuing with other I\+CU operations. ~\newline
 

The use of \mbox{\hyperlink{uclean_8h_a02cee2494284801dcc7f8bf7f12f0bcd}{u\+\_\+cleanup()}} just before an application terminates is optional, but it should be called only once for performance reasons. The primary benefit is to eliminate reports of memory or resource leaks originating in I\+CU code from the results generated by heap analysis tools. 

{\bfseries{Use this function with great care!}}

{\footnotesize\ttfamily \mbox{\hyperlink{umachine_8h_a676114b1a64bb7c2de15c919a00b28df}{U\+\_\+\+C\+A\+PI}} void U\+\_\+\+E\+X\+P\+O\+R\+T2 u\+\_\+init (\begin{DoxyParamCaption}\item[{\mbox{\hyperlink{utypes_8h_a3343c1c8a8377277046774691c98d78c}{U\+Error\+Code}} $\ast$}]{status }\end{DoxyParamCaption})}

Initialize I\+CU.

Use of this function is optional. It is OK to simply use I\+CU services and functions without first having initialized I\+CU by calling \mbox{\hyperlink{uclean_8h_a4774045efa17dd592f69f6988039a672}{u\+\_\+init()}}.

\mbox{\hyperlink{uclean_8h_a4774045efa17dd592f69f6988039a672}{u\+\_\+init()}} will attempt to load some part of I\+CU\textquotesingle{}s data, and is useful as a test for configuration or installation problems that leave the I\+CU data inaccessible. A successful invocation of \mbox{\hyperlink{uclean_8h_a4774045efa17dd592f69f6988039a672}{u\+\_\+init()}} does not, however, guarantee that all I\+CU data is accessible.

Multiple calls to \mbox{\hyperlink{uclean_8h_a4774045efa17dd592f69f6988039a672}{u\+\_\+init()}} cause no harm, aside from the small amount of time required.

In old versions of I\+CU, \mbox{\hyperlink{uclean_8h_a4774045efa17dd592f69f6988039a672}{u\+\_\+init()}} was required in multi-\/threaded applications to ensure the thread safety of I\+CU. \mbox{\hyperlink{uclean_8h_a4774045efa17dd592f69f6988039a672}{u\+\_\+init()}} is no longer needed for this purpose.
