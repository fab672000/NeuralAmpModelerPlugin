\doxysubsection{Detailed Description}
C A\+PI\+: Bidi algorithm. 

\doxysubsubsection*{Bidi algorithm for I\+CU}

This is an implementation of the Unicode Bidirectional Algorithm. The algorithm is defined in the \href{http://www.unicode.org/unicode/reports/tr9/}{\texttt{ Unicode Standard Annex \#9}}.

Note\+: Libraries that perform a bidirectional algorithm and reorder strings accordingly are sometimes called \char`\"{}\+Storage Layout Engines\char`\"{}. I\+CU\textquotesingle{}s Bidi and shaping (\mbox{\hyperlink{ushape_8h_a6316769ef467a9bb1d5da465d18663e5}{u\+\_\+shape\+Arabic()}}) A\+P\+Is can be used at the core of such \char`\"{}\+Storage Layout Engines\char`\"{}.

\doxyparagraph*{General remarks about the A\+PI\+:}

In functions with an error code parameter, the {\ttfamily p\+Error\+Code} pointer must be valid and the value that it points to must not indicate a failure before the function call. Otherwise, the function returns immediately. After the function call, the value indicates success or failure.

The "limit" of a sequence of characters is the position just after their last character, i.\+e., one more than that position.

Some of the A\+PI functions provide access to "runs". Such a "run" is defined as a sequence of characters that are at the same embedding level after performing the Bidi algorithm.

\doxysubparagraph*{Rendering a paragraph with the I\+CU Bidi A\+PI}

This is (hypothetical) sample code that illustrates how the I\+CU Bidi A\+PI could be used to render a paragraph of text. Rendering code depends highly on the graphics system, therefore this sample code must make a lot of assumptions, which may or may not match any existing graphics system\textquotesingle{}s properties.

The basic assumptions are\+:


\begin{DoxyItemize}
\item Rendering is done from left to right on a horizontal line. 
\item A run of single-\/style, unidirectional text can be rendered at once. 
\item Such a run of text is passed to the graphics system with characters (code units) in logical order. 
\item The line-\/breaking algorithm is very complicated and Locale-\/dependent -\/ and therefore its implementation omitted from this sample code. 
\end{DoxyItemize}

U\+B\+I\+D\+I\+\_\+\+O\+P\+T\+I\+O\+N\+\_\+\+S\+T\+R\+E\+A\+M\+I\+NG&option bit for {\ttfamily ubidi\+\_\+set\+Reordering\+Options}\+: process the output as part of a stream to be continued

This option must be set or reset before calling {\ttfamily ubidi\+\_\+set\+Para}.

This option specifies that the caller is interested in processing large text object in parts. The results of the successive calls are expected to be concatenated by the caller. Only the call for the last part will have this option bit off.

When this option bit is on, {\ttfamily \mbox{\hyperlink{ubidi_8h_acae95f0d8dd2e77386030462ec47afa5}{ubidi\+\_\+set\+Para()}}} may process less than the full source text in order to truncate the text at a meaningful boundary. The caller should call {\ttfamily \mbox{\hyperlink{ubidi_8h_a0a301c260f81a81640f5c9c19eb1226a}{ubidi\+\_\+get\+Processed\+Length()}}} immediately after calling {\ttfamily \mbox{\hyperlink{ubidi_8h_acae95f0d8dd2e77386030462ec47afa5}{ubidi\+\_\+set\+Para()}}} in order to determine how much of the source text has been processed. Source text beyond that length should be resubmitted in following calls to {\ttfamily ubidi\+\_\+set\+Para}. The processed length may be less than the length of the source text if a character preceding the last character of the source text constitutes a reasonable boundary (like a block separator) for text to be continued.~\newline
 If the last character of the source text constitutes a reasonable boundary, the whole text will be processed at once.~\newline
 If nowhere in the source text there exists such a reasonable boundary, the processed length will be zero.~\newline
 The caller should check for such an occurrence and do one of the following\+: 
\begin{DoxyItemize}
\item submit a larger amount of text with a better chance to include a reasonable boundary. 
\item resubmit the same text after turning off option {\ttfamily U\+B\+I\+D\+I\+\_\+\+O\+P\+T\+I\+O\+N\+\_\+\+S\+T\+R\+E\+A\+M\+I\+NG}.
\end{DoxyItemize}In all cases, this option should be turned off before processing the last part of the text.

When the {\ttfamily U\+B\+I\+D\+I\+\_\+\+O\+P\+T\+I\+O\+N\+\_\+\+S\+T\+R\+E\+A\+M\+I\+NG} option is used, it is recommended to call {\ttfamily \mbox{\hyperlink{ubidi_8h_a8e49e6589ecb494fc60cdf46d44e91cd}{ubidi\+\_\+order\+Paragraphs\+L\+T\+R()}}} with argument {\ttfamily order\+Paragraphs\+L\+TR} set to {\ttfamily true} before calling {\ttfamily ubidi\+\_\+set\+Para} so that later paragraphs may be concatenated to previous paragraphs on the right.

Set the context before a call to \mbox{\hyperlink{ubidi_8h_acae95f0d8dd2e77386030462ec47afa5}{ubidi\+\_\+set\+Para()}}.

\mbox{\hyperlink{ubidi_8h_acae95f0d8dd2e77386030462ec47afa5}{ubidi\+\_\+set\+Para()}} computes the left-\/right directionality for a given piece of text which is supplied as one of its arguments. Sometimes this piece of text (the \char`\"{}main text\char`\"{}) should be considered in context, because text appearing before (\char`\"{}prologue\char`\"{}) and/or after (\char`\"{}epilogue\char`\"{}) the main text may affect the result of this computation.

This function specifies the prologue and/or the epilogue for the next call to \mbox{\hyperlink{ubidi_8h_acae95f0d8dd2e77386030462ec47afa5}{ubidi\+\_\+set\+Para()}}. The characters specified as prologue and epilogue should not be modified by the calling program until the call to \mbox{\hyperlink{ubidi_8h_acae95f0d8dd2e77386030462ec47afa5}{ubidi\+\_\+set\+Para()}} has returned. If successive calls to \mbox{\hyperlink{ubidi_8h_acae95f0d8dd2e77386030462ec47afa5}{ubidi\+\_\+set\+Para()}} all need specification of a context, \mbox{\hyperlink{ubidi_8h_a0abab7d5bee13c6c5bbbaa799ed38e71}{ubidi\+\_\+set\+Context()}} must be called before each call to \mbox{\hyperlink{ubidi_8h_acae95f0d8dd2e77386030462ec47afa5}{ubidi\+\_\+set\+Para()}}. In other words, a context is not \char`\"{}remembered\char`\"{} after the following successful call to \mbox{\hyperlink{ubidi_8h_acae95f0d8dd2e77386030462ec47afa5}{ubidi\+\_\+set\+Para()}}.

If a call to \mbox{\hyperlink{ubidi_8h_acae95f0d8dd2e77386030462ec47afa5}{ubidi\+\_\+set\+Para()}} specifies U\+B\+I\+D\+I\+\_\+\+D\+E\+F\+A\+U\+L\+T\+\_\+\+L\+TR or U\+B\+I\+D\+I\+\_\+\+D\+E\+F\+A\+U\+L\+T\+\_\+\+R\+TL as para\+Level and is preceded by a call to \mbox{\hyperlink{ubidi_8h_a0abab7d5bee13c6c5bbbaa799ed38e71}{ubidi\+\_\+set\+Context()}} which specifies a prologue, the paragraph level will be computed taking in consideration the text in the prologue.

When \mbox{\hyperlink{ubidi_8h_acae95f0d8dd2e77386030462ec47afa5}{ubidi\+\_\+set\+Para()}} is called without a previous call to ubidi\+\_\+set\+Context, the main text is handled as if preceded and followed by strong directional characters at the current paragraph level. Calling \mbox{\hyperlink{ubidi_8h_a0abab7d5bee13c6c5bbbaa799ed38e71}{ubidi\+\_\+set\+Context()}} with specification of a prologue will change this behavior by handling the main text as if preceded by the last strong character appearing in the prologue, if any. Note 1\+: if {\ttfamily ubidi\+\_\+set\+Context} is called repeatedly without calling {\ttfamily ubidi\+\_\+set\+Para}, the earlier calls have no effect, only the last call will be remembered for the next call to {\ttfamily ubidi\+\_\+set\+Para}.

Modify the operation of the Bidi algorithm such that it approximates an \char`\"{}inverse Bidi\char`\"{} algorithm. This function must be called before {\ttfamily \mbox{\hyperlink{ubidi_8h_acae95f0d8dd2e77386030462ec47afa5}{ubidi\+\_\+set\+Para()}}}.

The normal operation of the Bidi algorithm as described in the Unicode Technical Report is to take text stored in logical (keyboard, typing) order and to determine the reordering of it for visual rendering. Some legacy systems store text in visual order, and for operations with standard, Unicode-\/based algorithms, the text needs to be transformed to logical order. This is effectively the inverse algorithm of the described Bidi algorithm. Note that there is no standard algorithm for this \char`\"{}inverse Bidi\char`\"{} and that the current implementation provides only an approximation of \char`\"{}inverse Bidi\char`\"{}.

With {\ttfamily is\+Inverse} set to {\ttfamily true}, this function changes the behavior of some of the subsequent functions in a way that they can be used for the inverse Bidi algorithm. Specifically, runs of text with numeric characters will be treated in a special way and may need to be surrounded with L\+RM characters when they are written in reordered sequence.

Output runs should be retrieved using {\ttfamily \mbox{\hyperlink{ubidi_8h_a9c095983e7f99a5999eb7d016783cd1e}{ubidi\+\_\+get\+Visual\+Run()}}}. Since the actual input for \char`\"{}inverse Bidi\char`\"{} is visually ordered text and {\ttfamily \mbox{\hyperlink{ubidi_8h_a9c095983e7f99a5999eb7d016783cd1e}{ubidi\+\_\+get\+Visual\+Run()}}} gets the reordered runs, these are actually the runs of the logically ordered output.

Calling this function with argument {\ttfamily is\+Inverse} set to {\ttfamily true} is equivalent to calling {\ttfamily ubidi\+\_\+set\+Reordering\+Mode} with argument {\ttfamily reordering\+Mode} set to {\ttfamily \mbox{\hyperlink{ubidi_8h_a88693e5a8ad4be974dc90ec6b8db56dfa6d23bae87c6e629866a3b87e11250e9a}{U\+B\+I\+D\+I\+\_\+\+R\+E\+O\+R\+D\+E\+R\+\_\+\+I\+N\+V\+E\+R\+S\+E\+\_\+\+N\+U\+M\+B\+E\+R\+S\+\_\+\+A\+S\+\_\+L}}}.~\newline

{\footnotesize\ttfamily \mbox{\hyperlink{umachine_8h_a676114b1a64bb7c2de15c919a00b28df}{U\+\_\+\+C\+A\+PI}} void U\+\_\+\+E\+X\+P\+O\+R\+T2 ubidi\+\_\+set\+Line (\begin{DoxyParamCaption}\item[{const \mbox{\hyperlink{ubidi_8h_aeed24292bbed966df93f088bc6791f74}{U\+Bi\+Di}} $\ast$}]{p\+Para\+Bi\+Di,  }\item[{int32\+\_\+t}]{start,  }\item[{int32\+\_\+t}]{limit,  }\item[{\mbox{\hyperlink{ubidi_8h_aeed24292bbed966df93f088bc6791f74}{U\+Bi\+Di}} $\ast$}]{p\+Line\+Bi\+Di,  }\item[{\mbox{\hyperlink{utypes_8h_a3343c1c8a8377277046774691c98d78c}{U\+Error\+Code}} $\ast$}]{p\+Error\+Code }\end{DoxyParamCaption})}

{\ttfamily \mbox{\hyperlink{ubidi_8h_af09b7e1254b150c95d3a048cf39b20f6}{ubidi\+\_\+set\+Line()}}} sets a {\ttfamily U\+Bi\+Di} to contain the reordering information, especially the resolved levels, for all the characters in a line of text. This line of text is specified by referring to a {\ttfamily U\+Bi\+Di} object representing this information for a piece of text containing one or more paragraphs, and by specifying a range of indexes in this text.

In the new line object, the indexes will range from 0 to {\ttfamily limit-\/start-\/1}.

This is used after calling {\ttfamily \mbox{\hyperlink{ubidi_8h_acae95f0d8dd2e77386030462ec47afa5}{ubidi\+\_\+set\+Para()}}} for a piece of text, and after line-\/breaking on that text. It is not necessary if each paragraph is treated as a single line.

After line-\/breaking, rules (L1) and (L2) for the treatment of trailing WS and for reordering are performed on a {\ttfamily U\+Bi\+Di} object that represents a line.

{\bfseries{Important\+: }}{\ttfamily p\+Line\+Bi\+Di} shares data with {\ttfamily p\+Para\+Bi\+Di}. You must destroy or reuse {\ttfamily p\+Line\+Bi\+Di} before {\ttfamily p\+Para\+Bi\+Di}. In other words, you must destroy or reuse the {\ttfamily U\+Bi\+Di} object for a line before the object for its parent paragraph.

The text pointer that was stored in {\ttfamily p\+Para\+Bi\+Di} is also copied, and {\ttfamily start} is added to it so that it points to the beginning of the line for this object.

{\footnotesize\ttfamily \mbox{\hyperlink{umachine_8h_a676114b1a64bb7c2de15c919a00b28df}{U\+\_\+\+C\+A\+PI}} void U\+\_\+\+E\+X\+P\+O\+R\+T2 ubidi\+\_\+set\+Para (\begin{DoxyParamCaption}\item[{\mbox{\hyperlink{ubidi_8h_aeed24292bbed966df93f088bc6791f74}{U\+Bi\+Di}} $\ast$}]{p\+Bi\+Di,  }\item[{const \mbox{\hyperlink{umachine_8h_a6bb9fad572d65b305324ef288165e2ac}{U\+Char}} $\ast$}]{text,  }\item[{int32\+\_\+t}]{length,  }\item[{\mbox{\hyperlink{ubidi_8h_ab2460a19f323ab9787a79a95db91a606}{U\+Bi\+Di\+Level}}}]{para\+Level,  }\item[{\mbox{\hyperlink{ubidi_8h_ab2460a19f323ab9787a79a95db91a606}{U\+Bi\+Di\+Level}} $\ast$}]{embedding\+Levels,  }\item[{\mbox{\hyperlink{utypes_8h_a3343c1c8a8377277046774691c98d78c}{U\+Error\+Code}} $\ast$}]{p\+Error\+Code }\end{DoxyParamCaption})}

Perform the Unicode Bidi algorithm. It is defined in the \href{http://www.unicode.org/unicode/reports/tr9/}{\texttt{ Unicode Standard Annex \#9}}, version 13, also described in The Unicode Standard, \mbox{\hyperlink{struct_version}{Version}} 4.\+0 .

This function takes a piece of plain text containing one or more paragraphs, with or without externally specified embedding levels from {\itshape styled} text and computes the left-\/right-\/directionality of each character.

If the entire text is all of the same directionality, then the function may not perform all the steps described by the algorithm, i.\+e., some levels may not be the same as if all steps were performed. This is not relevant for unidirectional text.~\newline
 For example, in pure L\+TR text with numbers the numbers would get a resolved level of 2 higher than the surrounding text according to the algorithm. This implementation may set all resolved levels to the same value in such a case.

The text can be composed of multiple paragraphs. Occurrence of a block separator in the text terminates a paragraph, and whatever comes next starts a new paragraph. The exception to this rule is when a Carriage Return (CR) is followed by a Line Feed (LF). Both CR and LF are block separators, but in that case, the pair of characters is considered as terminating the preceding paragraph, and a new paragraph will be started by a character coming after the LF.

\doxysubsubsection{\texorpdfstring{ubidi\_setReorderingMode()}{ubidi\_setReorderingMode()}}
{\footnotesize\ttfamily \mbox{\hyperlink{umachine_8h_a676114b1a64bb7c2de15c919a00b28df}{U\+\_\+\+C\+A\+PI}} void U\+\_\+\+E\+X\+P\+O\+R\+T2 ubidi\+\_\+set\+Reordering\+Mode (\begin{DoxyParamCaption}\item[{\mbox{\hyperlink{ubidi_8h_aeed24292bbed966df93f088bc6791f74}{U\+Bi\+Di}} $\ast$}]{p\+Bi\+Di,  }\item[{\mbox{\hyperlink{ubidi_8h_a88693e5a8ad4be974dc90ec6b8db56df}{U\+Bi\+Di\+Reordering\+Mode}}}]{reordering\+Mode }\end{DoxyParamCaption})}

Modify the operation of the Bidi algorithm such that it implements some variant to the basic Bidi algorithm or approximates an \char`\"{}inverse Bidi\char`\"{} algorithm, depending on different values of the \char`\"{}reordering mode\char`\"{}. This function must be called before {\ttfamily \mbox{\hyperlink{ubidi_8h_acae95f0d8dd2e77386030462ec47afa5}{ubidi\+\_\+set\+Para()}}}, and stays in effect until called again with a different argument.

The normal operation of the Bidi algorithm as described in the Unicode Standard Annex \#9 is to take text stored in logical (keyboard, typing) order and to determine how to reorder it for visual rendering.

With the reordering mode set to a value other than {\ttfamily \mbox{\hyperlink{ubidi_8h_a88693e5a8ad4be974dc90ec6b8db56dfac366446a9080bf062aca0080e9effc5b}{U\+B\+I\+D\+I\+\_\+\+R\+E\+O\+R\+D\+E\+R\+\_\+\+D\+E\+F\+A\+U\+LT}}}, this function changes the behavior of some of the subsequent functions in a way such that they implement an inverse Bidi algorithm or some other algorithm variants.

Some legacy systems store text in visual order, and for operations with standard, Unicode-\/based algorithms, the text needs to be transformed into logical order. This is effectively the inverse algorithm of the described Bidi algorithm. Note that there is no standard algorithm for this \char`\"{}inverse Bidi\char`\"{}, so a number of variants are implemented here.

In other cases, it may be desirable to emulate some variant of the Logical to Visual algorithm (e.\+g. one used in MS Windows), or perform a Logical to Logical transformation.


\begin{DoxyItemize}
\item When the reordering mode is set to {\ttfamily \mbox{\hyperlink{ubidi_8h_a88693e5a8ad4be974dc90ec6b8db56dfac366446a9080bf062aca0080e9effc5b}{U\+B\+I\+D\+I\+\_\+\+R\+E\+O\+R\+D\+E\+R\+\_\+\+D\+E\+F\+A\+U\+LT}}}, the standard Bidi Logical to Visual algorithm is applied.


\item When the reordering mode is set to {\ttfamily \mbox{\hyperlink{ubidi_8h_a88693e5a8ad4be974dc90ec6b8db56dfa447996fecd0eefb0d12ce247cb511e5f}{U\+B\+I\+D\+I\+\_\+\+R\+E\+O\+R\+D\+E\+R\+\_\+\+N\+U\+M\+B\+E\+R\+S\+\_\+\+S\+P\+E\+C\+I\+AL}}}, the algorithm used to perform Bidi transformations when calling {\ttfamily ubidi\+\_\+set\+Para} should approximate the algorithm used in Microsoft Windows XP rather than strictly conform to the Unicode Bidi algorithm. ~\newline
 The differences between the basic algorithm and the algorithm addressed by this option are as follows\+: 
\begin{DoxyItemize}
\item Within text at an even embedding level, the sequence \char`\"{}123\+A\+B\char`\"{} (where AB represent R or AL letters) is transformed to \char`\"{}123\+B\+A\char`\"{} by the Unicode algorithm and to \char`\"{}\+B\+A123\char`\"{} by the Windows algorithm. 
\item Arabic-\/\+Indic numbers (AN) are handled by the Windows algorithm just like regular numbers (EN). 
\end{DoxyItemize}


\item When the reordering mode is set to {\ttfamily \mbox{\hyperlink{ubidi_8h_a88693e5a8ad4be974dc90ec6b8db56dfa0cd75b3d1d8abc971554a2a7d459c6d7}{U\+B\+I\+D\+I\+\_\+\+R\+E\+O\+R\+D\+E\+R\+\_\+\+G\+R\+O\+U\+P\+\_\+\+N\+U\+M\+B\+E\+R\+S\+\_\+\+W\+I\+T\+H\+\_\+R}}}, numbers located between L\+TR text and R\+TL text are associated with the R\+TL text. For instance, an L\+TR paragraph with content \char`\"{}abc 123 D\+E\+F\char`\"{} (where upper case letters represent R\+TL characters) will be transformed to \char`\"{}abc F\+E\+D 123\char`\"{} (and not \char`\"{}abc 123 F\+E\+D\char`\"{}), \char`\"{}\+D\+E\+F 123 abc\char`\"{} will be transformed to \char`\"{}123 F\+E\+D abc\char`\"{} and \char`\"{}123 F\+E\+D abc\char`\"{} will be transformed to \char`\"{}\+D\+E\+F 123 abc\char`\"{}. This makes the algorithm reversible and makes it useful when round trip (from visual to logical and back to visual) must be achieved without adding L\+RM characters. However, this is a variation from the standard Unicode Bidi algorithm.~\newline
 The source text should not contain Bidi control characters other than L\+RM or R\+LM.


\item When the reordering mode is set to {\ttfamily \mbox{\hyperlink{ubidi_8h_a88693e5a8ad4be974dc90ec6b8db56dfa103b373c5933dd91cbfb2382b77bde8d}{U\+B\+I\+D\+I\+\_\+\+R\+E\+O\+R\+D\+E\+R\+\_\+\+R\+U\+N\+S\+\_\+\+O\+N\+LY}}}, a \char`\"{}\+Logical to Logical\char`\"{} transformation must be performed\+: 
\begin{DoxyItemize}
\item If the default text level of the source text (argument {\ttfamily para\+Level} in {\ttfamily ubidi\+\_\+set\+Para}) is even, the source text will be handled as L\+TR logical text and will be transformed to the R\+TL logical text which has the same L\+TR visual display. 
\item If the default level of the source text is odd, the source text will be handled as R\+TL logical text and will be transformed to the L\+TR logical text which has the same L\+TR visual display. 
\end{DoxyItemize}This mode may be needed when logical text which is basically Arabic or Hebrew, with possible included numbers or phrases in English, has to be displayed as if it had an even embedding level (this can happen if the displaying application treats all text as if it was basically L\+TR). ~\newline
 This mode may also be needed in the reverse case, when logical text which is basically English, with possible included phrases in Arabic or Hebrew, has to be displayed as if it had an odd embedding level. ~\newline
 Both cases could be handled by adding L\+RE or R\+LE at the head of the text, if the display subsystem supports these formatting controls. If it does not, the problem may be handled by transforming the source text in this mode before displaying it, so that it will be displayed properly.~\newline
 The source text should not contain Bidi control characters other than L\+RM or R\+LM.


\item When the reordering mode is set to {\ttfamily \mbox{\hyperlink{ubidi_8h_a88693e5a8ad4be974dc90ec6b8db56dfa6d23bae87c6e629866a3b87e11250e9a}{U\+B\+I\+D\+I\+\_\+\+R\+E\+O\+R\+D\+E\+R\+\_\+\+I\+N\+V\+E\+R\+S\+E\+\_\+\+N\+U\+M\+B\+E\+R\+S\+\_\+\+A\+S\+\_\+L}}}, an \char`\"{}inverse Bidi\char`\"{} algorithm is applied. Runs of text with numeric characters will be treated like L\+TR letters and may need to be surrounded with L\+RM characters when they are written in reordered sequence (the option {\ttfamily \mbox{\hyperlink{ubidi_8h_adad66f9132bc4e4621427091acfc0f40}{U\+B\+I\+D\+I\+\_\+\+I\+N\+S\+E\+R\+T\+\_\+\+L\+R\+M\+\_\+\+F\+O\+R\+\_\+\+N\+U\+M\+E\+R\+IC}}} can be used with function {\ttfamily ubidi\+\_\+write\+Reordered} to this end. This mode is equivalent to calling {\ttfamily \mbox{\hyperlink{ubidi_8h_ad071b1089276ba8044ae6e00b13271ad}{ubidi\+\_\+set\+Inverse()}}} with argument {\ttfamily is\+Inverse} set to {\ttfamily true}.


\item When the reordering mode is set to {\ttfamily \mbox{\hyperlink{ubidi_8h_a88693e5a8ad4be974dc90ec6b8db56dfa8ca4fe9b9d87e67c37c8a80bed3147aa}{U\+B\+I\+D\+I\+\_\+\+R\+E\+O\+R\+D\+E\+R\+\_\+\+I\+N\+V\+E\+R\+S\+E\+\_\+\+L\+I\+K\+E\+\_\+\+D\+I\+R\+E\+CT}}}, the \char`\"{}direct\char`\"{} Logical to Visual Bidi algorithm is used as an approximation of an \char`\"{}inverse Bidi\char`\"{} algorithm. This mode is similar to mode {\ttfamily \mbox{\hyperlink{ubidi_8h_a88693e5a8ad4be974dc90ec6b8db56dfa6d23bae87c6e629866a3b87e11250e9a}{U\+B\+I\+D\+I\+\_\+\+R\+E\+O\+R\+D\+E\+R\+\_\+\+I\+N\+V\+E\+R\+S\+E\+\_\+\+N\+U\+M\+B\+E\+R\+S\+\_\+\+A\+S\+\_\+L}}} but is closer to the regular Bidi algorithm. ~\newline
 For example, an L\+TR paragraph with the content \char`\"{}\+F\+E\+D 123 456 C\+B\+A\char`\"{} (where upper case represents R\+TL characters) will be transformed to \char`\"{}\+A\+B\+C 456 123 D\+E\+F\char`\"{}, as opposed to \char`\"{}\+D\+E\+F 123 456 A\+B\+C\char`\"{} with mode {\ttfamily U\+B\+I\+D\+I\+\_\+\+R\+E\+O\+R\+D\+E\+R\+\_\+\+I\+N\+V\+E\+R\+S\+E\+\_\+\+N\+U\+M\+B\+E\+R\+S\+\_\+\+A\+S\+\_\+L}.~\newline
 When used in conjunction with option {\ttfamily \mbox{\hyperlink{ubidi_8h_a4505e4adc8da792501414b770f49f386a23405ee9a1f1b254cc692132712ef9a6}{U\+B\+I\+D\+I\+\_\+\+O\+P\+T\+I\+O\+N\+\_\+\+I\+N\+S\+E\+R\+T\+\_\+\+M\+A\+R\+KS}}}, this mode generally adds Bidi marks to the output significantly more sparingly than mode {\ttfamily \mbox{\hyperlink{ubidi_8h_a88693e5a8ad4be974dc90ec6b8db56dfa6d23bae87c6e629866a3b87e11250e9a}{U\+B\+I\+D\+I\+\_\+\+R\+E\+O\+R\+D\+E\+R\+\_\+\+I\+N\+V\+E\+R\+S\+E\+\_\+\+N\+U\+M\+B\+E\+R\+S\+\_\+\+A\+S\+\_\+L}}} with option {\ttfamily \mbox{\hyperlink{ubidi_8h_adad66f9132bc4e4621427091acfc0f40}{U\+B\+I\+D\+I\+\_\+\+I\+N\+S\+E\+R\+T\+\_\+\+L\+R\+M\+\_\+\+F\+O\+R\+\_\+\+N\+U\+M\+E\+R\+IC}}} in calls to {\ttfamily ubidi\+\_\+write\+Reordered}.


\item When the reordering mode is set to {\ttfamily \mbox{\hyperlink{ubidi_8h_a88693e5a8ad4be974dc90ec6b8db56dfa0c426985f03689d0c4a2aade0e48c69e}{U\+B\+I\+D\+I\+\_\+\+R\+E\+O\+R\+D\+E\+R\+\_\+\+I\+N\+V\+E\+R\+S\+E\+\_\+\+F\+O\+R\+\_\+\+N\+U\+M\+B\+E\+R\+S\+\_\+\+S\+P\+E\+C\+I\+AL}}}, the Logical to Visual Bidi algorithm used in Windows XP is used as an approximation of an \char`\"{}inverse Bidi\char`\"{} algorithm. ~\newline
 For example, an L\+TR paragraph with the content \char`\"{}abc F\+E\+D123\char`\"{} (where upper case represents R\+TL characters) will be transformed to \char`\"{}abc 123\+D\+E\+F.\char`\"{} 
\end{DoxyItemize}

In all the reordering modes specifying an \char`\"{}inverse Bidi\char`\"{} algorithm (i.\+e. those with a name starting with {\ttfamily U\+B\+I\+D\+I\+\_\+\+R\+E\+O\+R\+D\+E\+R\+\_\+\+I\+N\+V\+E\+R\+SE}), output runs should be retrieved using {\ttfamily \mbox{\hyperlink{ubidi_8h_a9c095983e7f99a5999eb7d016783cd1e}{ubidi\+\_\+get\+Visual\+Run()}}}, and the output text with {\ttfamily \mbox{\hyperlink{ubidi_8h_a353e323fbd7d5dcee6e2ab50d6f1a3e4}{ubidi\+\_\+write\+Reordered()}}}. The caller should keep in mind that in \char`\"{}inverse Bidi\char`\"{} modes the input is actually visually ordered text and reordered output returned by {\ttfamily \mbox{\hyperlink{ubidi_8h_a9c095983e7f99a5999eb7d016783cd1e}{ubidi\+\_\+get\+Visual\+Run()}}} or {\ttfamily \mbox{\hyperlink{ubidi_8h_a353e323fbd7d5dcee6e2ab50d6f1a3e4}{ubidi\+\_\+write\+Reordered()}}} are actually runs or character string of logically ordered output.~\newline
 For all the \char`\"{}inverse Bidi\char`\"{} modes, the source text should not contain Bidi control characters other than L\+RM or R\+LM.

Note that option {\ttfamily \mbox{\hyperlink{ubidi_8h_a4a10c3aac68ceca1569bac717156cef3}{U\+B\+I\+D\+I\+\_\+\+O\+U\+T\+P\+U\+T\+\_\+\+R\+E\+V\+E\+R\+SE}}} of {\ttfamily ubidi\+\_\+write\+Reordered} has no useful meaning and should not be used in conjunction with any value of the reordering mode specifying \char`\"{}inverse Bidi\char`\"{} or with value {\ttfamily U\+B\+I\+D\+I\+\_\+\+R\+E\+O\+R\+D\+E\+R\+\_\+\+R\+U\+N\+S\+\_\+\+O\+N\+LY}.

Reverse a Right-\/\+To-\/\+Left run of Unicode text.

This function preserves the integrity of characters with multiple code units and (optionally) combining characters. Characters can be replaced by mirror-\/image characters in the destination buffer. Note that \char`\"{}real\char`\"{} mirroring has to be done in a rendering engine by glyph selection and that for many \char`\"{}mirrored\char`\"{} characters there are no Unicode characters as mirror-\/image equivalents. There are also options to insert or remove Bidi control characters.

This function is the implementation for reversing R\+TL runs as part of {\ttfamily \mbox{\hyperlink{ubidi_8h_a353e323fbd7d5dcee6e2ab50d6f1a3e4}{ubidi\+\_\+write\+Reordered()}}}. For detailed descriptions of the parameters, see there. Since no Bidi controls are inserted here, the output string length will never exceed {\ttfamily src\+Length}.
